❓ 分库分表常见问题

实际业务中最常遇到的问题与解决方案。

🔴
Q1：分表后别的服务怎么查询？需要也配置 ShardingSphere 吗？

需要。 ShardingSphere JDBC 是客户端分片，每个需要查询分片表的服务都要引入依赖并配置分片规则。 逻辑上你写的是 SELECT * FROM tb_order WHERE order_id=10， ShardingSphere 在 JDBC 层拦截、改写、路由到 tb_order_0，完全透明。

只需三步：① 引入 shardingsphere-jdbc-core-spring-boot-starter 依赖 ② 把 sharding.yml 复制到项目 resources 下 ③ application.yml 中 spring.config.import: classpath:sharding.yml

代码层面 完全不需要改，MyBatis/JPA 照常写，分片逻辑全在配置文件里。

🟡
Q2：分表后 ID 重复怎么办？还能用数据库自增吗？

不能用数据库自增！ tb_order_0 和 tb_order_1 各自自增，id 会重复。
解决方案：① ShardingSphere 雪花算法（推荐，配置 key-generators 即可，全局唯一） ② UUID ③ 自建 ID 生成服务（如百度 uid-generator、美团 Leaf）

在 sharding.yml 中配置后，@TableId(type = IdType.ASSIGN_ID) 或不设类型，MyBatisPlus 会用雪花ID。

🟢
Q3：分表后分页查询（LIMIT）怎么处理？

分片分页是著名的坑。ShardingSphere 会对每个分表执行 LIMIT，然后在内存中归并排序再截取。 例如 LIMIT 1000000, 10 会在内存中拉取 200万行然后丢掉前100万！
解决方案： ① 带分片键查询，让页码落在单表（最推荐） ② 游标分页（WHERE id > lastId LIMIT N），避免大偏移量 ③ 用 ES 做分页查询，DB 只用来查单条数据

🔵
Q4：需要单独的配置文件吗？怎么整合到已有项目？

强烈建议单独文件。 把分片规则放在 sharding.yml，在 application.yml 中用 spring.config.import: classpath:sharding.yml 引入。 这样分片配置和业务配置解耦，切换策略只改一行 import，不影响其他配置。
同时删掉 application.yml 里原来的 spring.datasource， 因为 ShardingSphere 会接管数据源。
